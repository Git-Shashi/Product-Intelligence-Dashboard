# Product Intelligence Dashboard

A full-stack e-commerce seller tool that extracts product data from videos or CSVs, validates listing quality, compares competitor prices, and raises severity-classified alerts. Built as a recruiter-facing take-home for Quantacus.

---

## Live Demo

| Service | URL |
|---|---|
| Frontend | https://product-intelligence-dashboard.vercel.app |
| Backend API | https://product-intelligence-api.onrender.com |
| API Docs (Swagger) | https://product-intelligence-api.onrender.com/docs |

## Running Locally

### System dependencies

```bash
# macOS
brew install ffmpeg tesseract

# Ubuntu / Render Docker
apt-get install -y ffmpeg tesseract-ocr tesseract-ocr-eng
```

### Backend

```bash
cd backend
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Copy and edit env
cp .env.example .env
# Edit DATABASE_URL to point at your local Postgres

# Run migrations
alembic upgrade head

# Seed demo data (optional but recommended)
python seed.py

# Start server
uvicorn app.main:app --reload
```

The API runs at `http://localhost:8000`. Swagger UI at `http://localhost:8000/docs`.

### Frontend

```bash
cd frontend
npm install
# Create .env.local
echo "VITE_API_URL=http://localhost:8000" > .env.local
npm run dev
```

App at `http://localhost:5173`.

## How to Use the Deployed App

1. **Upload** — go to the Upload page, drop the sample video (`samples/sample_product_video.mp4`) with the enhance-title toggle ON. The app extracts product fields with real OCR.
2. **Jobs** — watch job progress in real time. Completed jobs link to the product.
3. **Dashboard** — see quality score distribution, issue counts by severity, and weak listings.
4. **Products** — filter by severity / category. Click any product to see extracted fields, issues with suggested fixes, enhanced title, and competitor prices.
5. **Refresh Prices** — from the dashboard, hit Refresh Prices to re-fetch competitor prices and trigger alerts if we're priced >10% above the lowest competitor.
6. **Alerts** — severity-classified history. Acknowledge individually or mark all read.

## Data Model

```
products          — sku_id (unique), title, brand, category, price, mrp, image_url,
                    product_url, availability, color, size, material, source,
                    quality_score, created_at, updated_at

jobs              — type (VIDEO_EXTRACTION|CSV_VALIDATION|PRICE_REFRESH),
                    status (PENDING→RUNNING→COMPLETED/FAILED/PARTIALLY_COMPLETED),
                    progress 0–100, result_summary (JSONB with failed-row details)

listing_issues    — product_id (fk), type, severity (HIGH|MEDIUM|LOW),
                    message, suggested_fix

competitor_prices — product_id (fk), platform, competitor_url, competitor_price,
                    currency, last_checked_at

alerts            — product_id (fk, nullable), type, severity, message,
                    status (OPEN|ACKNOWLEDGED)

enhanced_titles   — product_id (fk), original_title, attributes (JSONB),
                    keywords (JSONB), enhanced_title, reason
```

All schema changes go through Alembic migrations (`alembic upgrade head`).

## Validation Rules

| Rule | Severity | Penalty |
|---|---|---|
| Missing title | HIGH | −20 |
| Short title (< 15 chars or < 3 words) | MEDIUM | −10 |
| Missing brand | MEDIUM | −10 |
| Invalid price (≤ 0 or missing) | HIGH | −20 |
| MRP lower than selling price | HIGH | −20 |
| Missing image URL | HIGH | −20 |
| Broken image URL (bad format) | MEDIUM | −10 |
| Missing ≥ 2 key attributes (color/size/material/category) | MEDIUM | −10 |
| Weak description (< 50 chars) | LOW | −3 |
| Out of stock | LOW | −3 |

**Quality score** = max(0, 100 − Σ penalties). A product with one HIGH and one MEDIUM issue scores 70.

---

## Alert Rules

| Condition | Severity |
|---|---|
| Missing title, no price, or invalid price | HIGH |
| Our price > 10% above lowest competitor | HIGH |
| Weak title or missing important attributes | MEDIUM |
| Competitor price dropped ≥ 5% since last refresh | MEDIUM |
| Weak description or out of stock | LOW |

On the initial product load (CSV / video upload) alerts are raised for ALL severities. On re-validation (edit / re-upload of same SKU) only HIGH alerts are re-raised to avoid noise.

## What Is Real vs Mocked

| Feature | Status |
|---|---|
| OCR extraction (ffmpeg + tesseract) | **Real** — extracts title, brand, price, color, size, material |
| Mock extractor fallback | Real — used when OCR yields nothing; seeded deterministically by SKU |
| Competitor prices | **Mocked** — 5 simulated platforms, jitter seeded by SKU MD5 |
| In-app notifications (alerts table) | **Real** — every alert row is a notification |
| Email notifications | Bonus — SMTP; silent no-op when `SMTP_HOST` is unset |

## Deployment

### Render (backend + Postgres)

1. Fork/push the repo to GitHub.
2. Go to Render → New Blueprint → select `render.yaml`.
3. Render auto-creates the web service (Docker) and the Postgres database.
4. Set `CORS_ORIGIN` to your Vercel URL once it's known.
5. The Dockerfile installs ffmpeg + tesseract-ocr at build time and hard-fails if either is missing, catching misconfigured base images before they reach production.

### Vercel (frontend)

1. Import the repo on Vercel. Set root directory to `frontend`.
2. Add env var: `VITE_API_URL=https://your-render-service.onrender.com`
3. Deploy.

After deploy, run the seed script once to populate demo data:

```bash
# SSH into the Render shell or use Render's one-off job
python seed.py
```

## Assumptions

- `sku_id` is the canonical product identifier. Uploading a CSV row with an existing SKU updates the product (upsert), not a duplicate error.
- Competitor prices are refreshed on demand, not on a schedule. Each refresh creates a new `competitor_prices` row, giving a natural price history per platform.
- "Broken image URL" validation checks URL format only (no network request); a MEDIUM rather than HIGH because the URL may be valid but temporarily unreachable.
- "Missing important attributes" fires when ≥ 2 of {color, size, material, category} are absent — a single missing attribute is common and not penalized separately.
- Email notifications are a bonus and require `SMTP_HOST` + `SMTP_USER` env vars; the app works fully without them.
- Quality score is additive-penalty from 100 (not weighted average) — simple to explain to sellers.
- The mock competitor service uses MD5(sku_id) as a seed so re-running returns the same prices deterministically. This makes test fixtures stable.

## Trade-offs and Limitations

- **No auth** — the assignment didn't require it; adding JWT middleware behind a feature flag would be the first step toward production hardening.
- **In-process background jobs** — FastAPI `BackgroundTasks` is simple but shares the server process. For production, swap the runner behind its interface with arq or Celery.
- **Competitor prices are mocked** — real price scraping requires platform-specific parsers and would need a scheduler (cron job or arq) rather than on-demand refresh.
- **OCR accuracy** — tesseract works well on high-contrast, standard-font text. Handwritten or stylized product videos degrade gracefully to the mock extractor.
- **Free Render tier** — cold starts after 15 minutes of inactivity. Not an issue for a demo; a paid tier or `render.yaml` health-check pings would fix it.

## What I'd Improve With More Time

1. **Price history chart** (Recharts) — every refresh already writes a timestamped row, so the data is there. Just needs a chart component on the product detail page.
2. **Scheduled price refresh** — arq or a cron job rather than manual button.
3. **Real competitor price APIs** — Flipkart Affiliate API or a lightweight scraper behind the `CompetitorService` protocol.
4. **Auth** — JWT + seller accounts so multiple sellers can use the same deployment.
5. **docker-compose** for local dev (postgres + backend + frontend in one `docker compose up`).
6. **Downloadable quality report** — CSV/PDF export of the dashboard summary.
7. **AI-vision extraction** — swap in GPT-4V or Gemini as an alternative `EXTRACTION_PROVIDER` for richer attribute parsing.
